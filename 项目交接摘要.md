## 1. 项目当前定位

当前项目是网页端朋友联机桌游 Demo，暂名《富饶之城》。

当前重点不是正式美术，也不是商业化，而是先跑通：

- 创建房间、加入房间
- 房间准备、人机补位、本机多身份测试
- 角色选择
- 按角色顺序行动
- 拿金币、抽牌、建造
- 8 个基础角色技能
- 隐藏信息过滤
- 离线 / 超时兜底
- 聊天
- 房间设置
- 完整打一局并结算

前端：React + TypeScript + Vite  
后端：Node.js + TypeScript + Express + Socket.IO  
数据：服务器内存状态，暂不接数据库

## 3. 最近已完成的重要功能

### 房间系统

- 创建房间 / 加入房间
- 房主标识
- 准备 / 取消准备
- 添加测试人机
- 删除测试人机
- 房主踢人
- 房主移交
- 房主离开后自动移交
- 房间聊天
- 房间设置弹窗
- 房间公开设置展示

### 开局倒计时

- 所有人准备后触发 10 秒自动开始倒计时
- 但必须至少有 2 名在线真人玩家
- 只有房主 + 人机时，不会自动倒计时
- 手动开始游戏仍允许房主 + 测试人机，用于单人测试

### 游戏核心

- 游戏开始后发初始金币和手牌
- 进入角色选择阶段
- 按王冠玩家开始选角
- 选角完成后按角色 `order` 行动
- 一轮结束后进入下一轮
- 国王可以改变下一轮选角先手
- 当前 MVP 可以完整打完一局

### 隐藏信息

- 每个玩家只能看到自己的手牌
- 看不到其他玩家未公开角色
- 看不到建筑牌堆顺序
- 角色选择阶段只有当前选角玩家能看到 `availableRoles`

### 离线 / 超时

- 玩家断线不立即删除游戏内玩家
- 游戏中玩家离线会标记为离线
- 当前行动玩家离线时，可跳过 / 托管推进
- 回合倒计时由服务端状态提供，前端只展示和请求处理

## 4. 当前规则审查结论

当前代码不是严格标准包正式规则，而是：

> 8 个标准角色 + MVP 简化规则

已经明确的流程：

1. 房间开始游戏
2. 每人获得初始金币和手牌
3. 进入角色选择
4. 从王冠玩家开始依次秘密选角色
5. 选完后进入角色行动
6. 按角色编号 1 到 8 依次行动
7. 当前玩家执行资源行动、建造、技能、结束回合
8. 所有角色行动完成后进入下一轮
9. 达到建造数量后结算

但和标准包规则相比，当前存在这些简化 / 缺口：

- 结束建筑数当前是 `4`，不是标准常见的 `8`
- 抽牌当前是“抽 2 全拿”，不是“抽 2 选 1”
- 角色选择没有正式弃牌规则
- 2 / 3 / 4 人局每人选几个角色还没有正式配置化
- 房间没有配置“本局启用哪些角色”
- 房间没有配置“是否明弃 / 暗弃角色”
- 房间没有配置“结束建筑数量”
- 角色收入没有补齐：
  - 国王：黄色建筑收入未补
  - 主教：蓝色建筑收入未补
  - 商人：绿色建筑收入已有，但固定 +1 金币未补
  - 军阀：红色建筑收入未补
- 角色技能未完全标准：
  - 魔术师缺“与另一名玩家交换手牌”
  - 建筑师缺“额外抽 2 张牌”
  - 军阀需确认是否禁止破坏已完成城市玩家的建筑
- 建造同名建筑不可重复的校验，文档写了但当前代码里未看到实际校验
- 计分还很简化，只有建筑分 + 建满奖励，没有完整五色奖励 / 首个建满奖励 / 紫色建筑特殊计分

## 5. 用户最新规则意图

用户明确提出：

1. 规则相关参数需要写入房间设置。
2. 默认配置按 4 人规则处理。
3. 抽牌必须改成“抽 2 选 1”。
4. 角色收入和技能要补齐到标准版。
5. 默认使用 8 个角色。
6. 房主可以在房间设置里配置本局启用哪些角色。
7. 是否加入明弃 / 暗弃角色，写入房间设置。
8. 如果设置启用明弃 / 暗弃，游戏中就执行；不启用就不执行。
9. 结束条件写入房间设置。
10. 房间界面需要公示：
    - 结束条件
    - 本局启用哪些角色
    - 是否明弃
    - 是否暗弃
    - 每轮等待时间等公开参数

## 6. 建议下一步设计方向

建议不要马上直接改代码，而是先确认并落地一个规则配置方案。

推荐新增阶段：

### 阶段 25B：标准包 8 角色规则配置化与规则口径锁定

目标：

- 把“标准包规则”和“测试便利选项”统一收进 `RoomSettings`
- 默认按 4 人标准包配置
- 房间界面公示关键规则
- 为后续 2-8 人扩展、进阶角色、UI 替换留接口

建议新增 / 扩展配置：

```ts
type RoomSettings = {
  startCountdownSeconds: number;
  turnTimeoutSeconds: number;

  endCitySize: number;              // 默认 8
  enabledRoleIds: string[];         // 默认 8 个标准角色
  enableFaceUpRoleDiscard: boolean; // 是否明弃
  enableFaceDownRoleDiscard: boolean; // 是否暗弃
  roleSelectionMode: "oneRoleEach" | "twoRolesEach" | "standardByPlayerCount";
  drawMode: "draw2Keep1";
  enableRoleColorIncome: boolean;   // 默认 true
  enableStandardRoleSkills: boolean; // 默认 true
};
```

注意：`drawMode` 目前用户已明确要求必须是 `draw2Keep1`，可以先不做其他模式。

## 7. 需要进一步确认的问题

这些问题建议新对话里先问用户，避免规则做错：

1. 4 人默认规则中，每人是只选 1 个角色，还是按正式 4 人规则每人可能选 2 个角色？
2. 明弃 / 暗弃的数量是否要跟随玩家人数自动计算，还是房主直接填数量？
3. 当前仍需要支持“1 真人 + 人机手动开局”测试吗？
4. 如果启用 8 建筑结束条件，单人测试一局会明显变长，是否保留“测试模式结束建筑数”快捷配置？
5. 军阀是否允许破坏已经建满城市玩家的建筑？
6. 计分是否现在一起补齐，还是先只补抽牌、收入、角色技能？

## 10. 当前开发原则

- 服务端是权威状态
- 前端不能自己决定游戏结果
- 房间设置必须由服务端校验
- 动态隐藏信息不能发给不该看到的玩家
- UI 美术先放后，规则稳定后再正式接入
- 每次阶段变更都要同步 `制作序列.txt`
- 不要大重构，优先小步推进、可测试、可回退
