# 项目结构归纳规则

本文档记录本项目的长期结构规则。换新对话后，开始改动前应先阅读本文件。后续如果新增功能、调整 UI 流程、拆分模块或发现更好的归类方式，可以同步更新本文档。

## 当前目标

本项目按正式版游戏结构逐步整理。当前优先稳定：

- 大厅
- 准备房间
- 对局
- 结算

目前大厅、准备房间、对局和结算界面都已经开始按游戏阶段拆分。
## 当前大厅 / 准备房间拆分

当前大厅和准备房间相关文件集中在 `client/src/pages/lobby`。

- `HomeMenu.tsx`：大厅主入口，负责创建房间和输入房间号加入。
- `ReadyRoom.tsx`：准备房间界面，负责房间信息、座位、人机、聊天和准备操作的显示。
- `RoomSettingsModal.tsx`：房间设置弹窗。
- `InfoModal.tsx`：大厅和房间共用的信息弹窗。
- `PlayerIdentity.tsx`：左上角玩家头像、昵称和上传头像入口。
- `LobbyFooter.tsx`：公告、帮助、设置、退出房间和版本号底部栏。
- `useRoomSettings.ts`：房间设置表单状态、弃牌显示、角色数量校验。
- `useLobbyRoom.ts`：创建房间、加入房间、离开房间、添加人机、座位操作、房间设置保存等房间动作。

`ConnectionPage.tsx` 保留为当前大厅流程容器，主要负责服务器连接状态、页面切换和把状态传给各个界面模块。后续不要再把新的大厅/房间细节直接堆进 `ConnectionPage.tsx`，应优先放进上面的对应文件。

## 当前对局拆分

当前对局界面相关文件集中在 `client/src/pages/game`。

- `GameTableView.tsx`：对局界面总容器，接收游戏状态、动作回调和确认弹窗。
- `GameTopBar.tsx`：房间号、轮次、阶段和公告 / 帮助 / 设置 / 退出入口。
- `GameCenterStatus.tsx`：桌面中心的游戏标题、当前阶段、等待提示和倒计时。
- `GameOpponentSeat.tsx`：对手座位、头像、金币、手牌数、建筑数量和小牌展示。
- `GameSelfArea.tsx`：自己的牌堆、玩家信息、身份牌、手牌和弃牌堆。
- `GameSelfCity.tsx`：自己的已建建筑展示层，独立排列在中央状态与行动栏之间。
- `GameObjectiveNotice.tsx`：开局胜利条件展示，以及左上目标摘要文案。
- `DistrictChoiceCard.tsx`：抽卡二选一 / 三选一使用的完整建筑卡牌组件，并预留正式卡图区域。
- `GameCardInspector.tsx`：对局内角色牌、手牌、抽牌候选和公开建筑共用的悬停检视器，负责 50ms 关闭延迟和视口边缘自适应定位。
- `GameSkillPresentationLayer.tsx`：对局关键技能的独立桌面演出覆盖层，根据玩家标签的实时 DOM 位置绘制技能路径、目标光圈、卡牌 / 金币移动和结果横幅，不参与座位与手牌布局。
- `GameActionNoticeLayer.tsx`：与移动动画解耦的文字结果层，最多显示两条并按表现配置保留阅读时间。
- `presentationTiming.ts`：金币 / 抽卡、普通行动、技能和阶段提示共用的唯一时长配置；CSS 动画必须读取这里下发的变量。
- `useGameCommandFeedback.ts`、`GameCommandFeedbackToast.tsx`：对局命令的处理中、超时解锁和失败原因反馈，不把 Socket 确认逻辑散落进按钮组件。
- `gameUiTuning.ts`、`GameUiTuningPanel.tsx`：V2 调参配置、人数密度预设、V1 迁移和组件间安全值计算；正式环境不注册调参入口。
- `cardInspectorData.ts`：把角色 / 建筑公开信息转换为统一检视器数据属性，禁止把未公开身份或对手私有手牌送入检视器。
- `GameActionDock.tsx`：角色选择、抽牌选择、行动阶段按钮、角色技能和建筑效果入口。
- `GameCornerDocks.tsx`：游戏日志和聊天入口。
- `gameTableLayout.ts`：4 到 8 人对局座位位置分配。

大厅、准备房间与对局右上角共用 `client/src/components/ui/UtilityMenuButton.tsx`。公告、帮助、设置和退出房间必须从该共享组件进入，避免各阶段再次复制或用 CSS 临时绘制不同图标。

对局模块不要依赖大厅和准备房间的 UI 结构。它只接收房间和游戏状态，然后显示当前对局。后续新增对局功能时，优先放入这个目录，再根据复用情况决定是否抽到通用组件。

## 总体原则

- 不把新功能随手塞进大页面。
- 新功能先判断属于哪个游戏阶段，再放入对应模块。
- UI 显示、状态控制、服务器连接、游戏规则尽量分开。
- 前后端都要用的规则放到 `shared`。
- 只由服务端裁定的规则放到 `server/src/game`。
- 背景、美术资源、可替换 UI 资源要保持路径清晰，方便后续换皮。
- 如果某个文件开始同时负责太多事情，优先拆出小模块，而不是继续堆逻辑。

## 对局可靠性与验收归类

- 前后端共用的命令确认、日志来源和表现事件字段放在 `shared/src/index.ts`；服务端仍是唯一结算方。
- 自动操作必须使用 `player / bot / timeout / offline / rule` 来源，并通过 `server/src/game/gameEngineUtils.ts` 的上下文写入日志和表现事件。
- 卡牌守恒检查集中在 `server/src/game/cardIntegrity.ts`，规则覆盖和 4–8 人固定种子模拟放在同目录测试中。
- UI 极端场景只能通过 `ZY_ENABLE_UI_QA=1` 注册，生产环境不得暴露 `qa_configure_game`。
- `scripts/run-regression.mjs` 负责快速与完整报告；本次启动的服务、页面和浏览器必须在 `finally` 中关闭，不能在桌面留下递归预览窗口。
- `scripts/smoke-server-restart-recovery.mjs` 必须真实停止并重启服务，验证房间快照、恢复令牌和恢复后继续行动。
- 正式卡图使用固定同名目录与 `VITE_CARD_FACE_MODE`，资源缺失必须回退程序卡面并由严格美术检查报告，不得伪装为已交付。

## 前端归类

### 大厅

大厅相关功能放在 `client/src/pages/lobby` 下。大厅只负责进入游戏前的主入口体验。

适合放入大厅模块的内容：

- 创建房间
- 输入房间号加入房间
- 玩家基础展示
- 大厅按钮
- 大厅背景和主视觉
- 公告、帮助、设置入口

大厅界面组件应尽量只负责显示，不直接承载大量服务器事件逻辑。

### 准备房间

准备房间相关功能也放在 `client/src/pages/lobby` 下，直到后续需要单独拆成 `room` 目录。

适合放入准备房间模块的内容：

- 房间号展示
- 玩家座位
- 添加人机
- 添加座位
- 删除空座位
- 开始游戏
- 离开房间
- 房间设置入口
- 聊天框
- 准备状态
- 房间规则摘要

准备房间应重点保证 4 到 8 人扩展稳定。4 人时不需要强制出现滚动，5 到 8 人时再允许座位区滚动或扩展布局。

### 房间设置

房间设置放在独立组件中，不直接写进准备房间主体。

适合放入房间设置的内容：

- 回合时间
- 结束条件
- 本局弃牌规则
- 角色启用规则
- 其他只影响房间规则的配置

如果某个设置会受到人数影响，例如明弃 / 暗弃是否可用，应由共享规则计算结果，再让 UI 根据结果变灰或禁用。

### 对局

对局界面当前放在 `client/src/pages/game`。后续对战 UI、座位布局、手牌区、操作层和阶段提示优先在这里继续拆。

适合放入对局模块的内容：

- 玩家区域
- 手牌区域
- 建筑区域
- 角色选择
- 当前行动面板
- 回合提示
- 游戏日志
- 技能操作
- 4 到 8 人的对局座位布局

对局模块不要依赖大厅和准备房间的 UI 结构。它只接收房间和游戏状态，然后显示当前对局。

### 结算

正式结算界面放在 `client/src/pages/result`，结算专用样式放在 `client/src/styles/result-screen.css`。

- `GameResultOverlay.tsx`：结算总容器、胜者玩家标签、排名和返回大厅入口。
- `ResultDistrictCard.tsx`：结算页面使用的只读建筑牌，完整展示每位玩家已建城市。

适合放入结算模块的内容：

- 玩家排名
- 分数明细
- 奖励展示
- 返回大厅
- 再来一局
- 对局记录

结算逻辑和显示应分开。服务端负责分数与胜负结果，前端负责表现。

### 通用 UI

多个界面都会使用的内容应放入通用组件，而不是复制到每个界面。

适合归入通用 UI 的内容：

- 头像
- 通用按钮
- 通用弹窗
- 图标按钮
- 状态提示
- 轻量消息提示
- 可复用面板
- 资源按钮

如果组件只在大厅 / 准备房间出现，可以先留在 `client/src/pages/lobby`；当第二个大模块也需要它时，再移动到通用组件目录。

## 样式与资源

大厅和准备房间的样式当前集中在 `client/src/styles/lobby-screens.css`。

样式规则：

- 页面级布局可以放在对应页面样式文件。
- 通用主题变量放在主题样式中。
- 不要让装饰线条压到 UI 边框。
- 背景不能抢信息层级，中间内容应最突出。
- 面向 12 到 18 岁玩家时，风格可以更明亮、更风格化，但不要过度杂乱。

背景资源规则：

- 大厅背景和准备房间背景要分开。
- 背景路径应通过明确的 CSS 变量或配置引用，方便替换。
- 新生成背景应放在 `client/public/assets/generated-ui` 或后续明确的资源目录。
- 对局正式背景应在玩家座位、中央状态、建筑区、行动栏和手牌区的布局全部确认后再生成。
- 对局背景生成时应按最终 UI 安全区域构图，保留中央操作空间和四周玩家信息空间，背景本身不绘制不可替换的按钮、文字或卡牌。

## 共享规则

前后端都要知道的规则放在 `shared/src/index.ts` 或后续拆出的共享规则文件。

适合放入共享规则的内容：

- 人数上限
- 默认座位数
- 角色弃牌规则
- 房间设置类型
- 前后端共同使用的数据结构

例如明弃 / 暗弃规则需要前端显示变灰，也需要服务端真正执行，因此应保持在共享层。

## 服务端规则

只由服务端裁定和保存的游戏逻辑放在 `server/src/game`。

适合放入服务端规则的内容：

- 创建房间
- 加入房间
- 离开房间
- 添加人机
- 开始游戏
- 回合流程
- 角色技能
- 建筑效果
- 计分
- 胜负判定
- 掉线和重连状态

前端不能成为权威裁判。前端只负责发送玩家操作和显示服务端结果。

对建筑执行破坏时，角色技能与后续可能复用该规则的服务端逻辑统一经过
`server/src/game/districtDestruction.ts`。目标是否存在、城市保护、完整城市、要塞、长城加价、金币扣除和墓地收回都由服务端最终裁定，前端高亮只用于操作提示。

## 后续新增功能的归纳流程

新增功能时，先按下面顺序判断：

1. 这是大厅、准备房间、对局还是结算功能？
2. 它只是 UI 显示，还是会改变游戏规则？
3. 它是否需要前后端都知道？
4. 它是否应该成为通用组件？
5. 它是否会影响现有结构规则？

判断后再落位：

- 只影响显示：放前端对应界面模块。
- 影响房间或对局状态：前端显示 + 服务端规则都要改。
- 前后端都需要同一条规则：放 `shared`。
- 多个界面复用：抽成通用 UI。
- 会改变项目归类方式：同步更新本文档。

## 当前建议的演进节奏

当前阶段：

- 稳定大厅和准备房间。
- 保持 4 到 8 人房间逻辑可用。
- 保持房间设置、弃牌规则和座位逻辑清晰。
- 不急着做全游戏最终架构。

下一阶段：

- 对局模块已开始拆出，后续先稳定对战布局、阶段操作和 4 到 8 人座位，再拆结算模块。
- 对局模块稳定后，再拆结算模块。
- 当大厅、准备房间、对局、结算都稳定后，再整理更完整的正式版目录。

## 规则迭代

本文档不是一次性固定文件。后续如果出现以下情况，应更新本文档：

- 新增了重要目录。
- 某类功能有了更准确的归属。
- 大页面被进一步拆分。
- 对局或结算结构确定。
- UI 资源管理方式改变。
- 共享规则被拆成更细文件。
- 用户明确提出新的长期开发习惯。

更新规则时，要保持文字清楚，避免写成只适合当前一次任务的临时说明。

## 对局可见状态与隐藏信息

- `GameRoom` 是服务端内部完整状态，不能直接整体发送给客户端。
- 所有 `game_state` 广播必须先经过 `server/src/game/visibility.ts`，按查看者生成 `VisibleGameState`。
- 对手手牌只能公开数量；未揭示身份只能公开“未公开”状态。
- 私有手牌进入弃牌堆后，客户端只接收弃牌数量，不能接收完整卡牌对象。
- 当前身份行动顺序只公开已经叫到或已经完成的身份，不能提前发送整轮已选身份集合。
- 新增技能、建筑效果或牌堆时，必须同时检查网络可见状态并补充隐藏信息测试。

## 对局王冠与叫号

- 中央角色行动提示使用当前已公开身份，格式为“编号号身份名 · 玩家名行动”；角色选择阶段不能提前显示身份。
- 玩家名片上的王冠只依据服务端公开的 `crownPlayerId`，前端不能根据秘密身份自行推断。
- 国王身份正常被叫到时转移王冠；如果国王被刺客跳过，则在本轮结束后再转移，避免泄露秘密身份。
- 正式对局背景当前使用 `client/public/assets/generated-ui/citadels-game-table-background-v2.png`，通过版本化文件名保持可替换并避免浏览器缓存旧图。

## 对局建筑效果与结算

- 主动建筑效果从 `GameSelfCity.tsx` 中已建好的建筑牌进入，不再在通用行动栏增加“建筑效果”二级入口。
- 需要弃牌的建筑效果使用客户端本地选择态：手牌区负责选择，手牌右侧负责确定或放弃；只有确定时才向服务端发送牌 ID，不能把私有选择写入公开 `TurnState`。
- 抽牌二选一未完成时，服务端必须锁定建造、角色技能和建筑效果；结束回合可以放弃候选牌，并自动领取默认的 2 枚金币。
- 正式结算界面放在 `client/src/pages/result`，展示服务器计分结果，并按 `playerId` 关联公开城市数据，完整展示每位玩家的已建建筑。
- 结算页样式放在 `client/src/styles/result-screen.css`，4 到 8 人统一使用纵向排名滚动，单个玩家的建筑牌轨道可以横向滚动。
- 正式王冠资源使用 `client/public/assets/generated-ui/citadels-crown-icon-v1.png`，由玩家名片组件放在头像上方；资源本身保持透明且不带外框。
- 需要选择牌桌卡牌的主动效果直接在对应公开区域选择：军阀在其他玩家的公开城市中选择建筑，实验室在自己的手牌区选择弃牌；不使用默认下家或隐藏下拉框代替牌桌目标选择。
- 角色技能的临时目标交互放在 `client/src/pages/game/roleSkillTargeting.ts`；身份目标、自己手牌目标和其他玩家目标都必须由玩家显式选择，不能在视图模型中默认第一项。
- 需要其他玩家即时回应的公开建筑效果使用共享 `GameRoom` pending choice 状态、服务端解析函数和客户端确认界面；pending choice 必须锁定会产生冲突的行动，并提供机器人、离线与超时处理。
- 对局内卡牌说明统一经过 `GameCardInspector.tsx`：角色牌、自己的手牌、抽牌候选、自己已建建筑和对手公开建筑复用同一套轻量预览；对手隐藏手牌和未公开身份只能展示已有的公开牌背信息。

## 任务执行效率规则

- 默认由主 Agent 直接检查、修改和验证，不为普通 UI 调整强制启动多 Agent。
- 只有子任务彼此真正独立、预计耗时较长，或需要单独进行隐藏信息 / 安全审查时才使用 subagent。
- 单次任务原则上最多启动 1 到 2 个 subagent；短任务、同文件任务和需要频繁整合的任务不并行。
- UI 调整优先采用“读取相关模块 -> 主 Agent 修改 -> 单个代表性视口验收 -> 构建检查”的轻量流程；完整多分辨率矩阵留到阶段性定稿。
- 不重复扫描已确认的项目范围，不让多个 Agent 输出相同类型的分析，不为了形式上的分工增加 token 和等待时间。
