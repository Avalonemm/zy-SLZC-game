# 角色技能规则设计表

这份文档用于后续设计《富饶之城》的 MVP 技能、进阶版技能和可能的技能动画事件。当前先作为规则参考，不代表最终正式规则。

## 设计原则

1. 服务端永远是技能结算的权威。
2. 前端只负责展示可选目标和发送技能请求。
3. 技能是否可用、目标是否合法、效果是否生效，都由服务端判断。
4. 动画只能响应服务端确认后的结果，不能决定游戏结果。
5. 每个技能都要写入 gameLog，方便玩家理解发生了什么。
6. 复杂技能后续需要拆成：目标选择、合法性校验、效果结算、事件通知、动画表现。

## 技能字段模板

| 字段 | 说明 |
| --- | --- |
| 角色 | 角色名称 |
| 技能类型 | 主动 / 被动 / 触发型 / 延迟结算 |
| 使用时机 | 角色选择阶段 / 自己行动时 / 目标行动前 / 建造前 / 建造后 / 回合结束时 |
| 目标类型 | 无 / 角色 / 玩家 / 建筑 / 手牌 / 牌堆 |
| 目标数量 | 0 / 1 / 多个 / 任意数量 |
| 合法性限制 | 不能选自己、不能选已行动角色、不能选受保护建筑等 |
| 效果结算 | 实际改变哪些服务端状态 |
| 日志 | 需要写入的玩家可读说明 |
| 动画事件 | 后续可选，用于前端播放技能表现 |
| 边界情况 | 目标消失、金币不足、牌堆不足、离线玩家、人机行动等 |

## MVP 当前 8 个角色

| 角色 | 技能类型 | 使用时机 | 目标类型 | 当前效果 | 当前边界 |
| --- | --- | --- | --- | --- | --- |
| 刺客 | 主动 / 延迟结算 | 自己行动时指定，目标角色行动前生效 | 角色 | 指定一个角色本轮跳过行动 | 不能指定自己，不能指定已行动角色 |
| 盗贼 | 主动 / 延迟结算 | 自己行动时指定，目标角色行动前生效 | 角色 | 目标角色行动前，盗贼获得该玩家全部金币 | 不能偷刺客，不能偷自己，不能偷被刺杀角色，不能偷已行动角色 |
| 魔术师 | 主动 | 自己行动时 | 手牌 | 弃置任意已选手牌，然后抽等量建筑牌 | 必须选择自己手牌中存在的牌，牌堆不足时抽剩余数量 |
| 国王 | 主动 | 自己行动时 | 无 | 获得下一轮角色选择先手权 | 每回合只能使用一次 |
| 主教 | 主动 / 防护 | 自己行动时 | 无 | 本轮自己的城市不能被军阀破坏 | 每回合只能使用一次 |
| 商人 | 主动 / 收益 | 自己行动时 | 无 | 按自己绿色建筑数量获得金币 | 绿色建筑为 0 时获得 0 金币，但仍算使用技能 |
| 建筑师 | 主动 / 强化 | 自己行动时 | 无 | 本回合额外获得 1 次建造机会 | 只影响当前 turnState |
| 军阀 | 主动 | 自己行动时 | 玩家 + 建筑 | 花费目标建筑费用 - 1 的金币，破坏其他玩家一座建筑 | 不能破坏自己建筑，不能破坏受主教保护玩家的建筑，金币不足不能破坏 |

## 前端目标选择结构

当前前端测试界面已将技能目标选择整理为统一结构，代码入口为 `client/src/components/test-game/skillTargeting.ts`。

| 目标结构 | 当前角色 | 前端展示 |
| --- | --- | --- |
| none | 国王、主教、商人、建筑师 | 不显示目标选择，直接允许使用技能 |
| role | 刺客、盗贼 | 显示角色下拉选择 |
| discardCards | 魔术师 | 显示自己的手牌勾选 |
| district | 军阀 | 显示目标玩家和目标建筑选择 |

后续新增进阶角色时，优先在目标结构中增加或复用目标类型，再让服务端技能校验跟进；不要把某个角色的目标 UI 临时写死在 ActionPanel 里。

## 后续进阶技能预留方向

| 类型 | 可能需求 | 代码预留方向 |
| --- | --- | --- |
| 多目标技能 | 同时选择多个玩家、多个建筑、多个手牌 | 统一 target payload，支持数组目标 |
| 被动技能 | 建造时触发、收入时触发、结算时触发 | 增加技能触发点，而不是只靠 use_role_skill |
| 反应技能 | 其他玩家行动时可响应 | 增加可响应窗口、超时或自动跳过策略 |
| 延迟技能 | 本轮稍后结算或下轮结算 | roleEffects 保存待结算效果 |
| 动画事件 | 偷金币、破坏建筑、换牌、获得王冠等表现 | 服务端返回 action_event，前端按事件播放动画 |
| 目标选择 UI | 角色、玩家、建筑、手牌、牌堆选择不同 | 前端根据技能元数据渲染目标选择控件 |

## 后续实现提醒

1. 不要把技能规则写死在 UI 组件里。
2. 不要让前端根据按钮直接改金币、手牌、建筑。
3. 每新增一个技能，至少补一条服务端测试。
4. 如果技能需要动画，先确认服务端状态，再由前端播放。
5. 如果技能会打断当前流程，需要先设计状态机阶段或响应窗口。
