## 1. 当前目标

项目正在把《富饶之城》原型整理为正式版结构。当前界面只保留四个阶段：

- 大厅
- 准备房间
- 对局
- 结算

首页不需要。当前优先级是继续稳定对局 UI、4 到 8 人布局和正式游戏流程；不要重新推翻已经确认的大厅与准备房间布局。

## 2. 工作区保护要求

- 当前工作区存在大量未提交修改，其中包含用户已经确认的有效成果。
- 不要执行 `git reset --hard`、`git checkout --` 或批量回滚。
- 不要因为文件未跟踪就删除它们；当前新拆出的大厅、对局、结算模块和生成资源大多仍是未跟踪文件。
- 修改前先读取目标文件现状，在现有成果上继续迭代。
- 不要为普通 UI 调整重新启动大规模多 Agent 流程。

### 当前游戏流程

- 如果玩家没有领取金币或抽牌就结束回合，服务端会自动选择领取 2 金币。
- 抽牌候选未完成时会锁定建造、角色技能和建筑效果。
- 建造确认弹窗在确认、取消或回合结束后会关闭。
- 人机会正常进行资源选择和建造，不应只结束回合。
- 主动建筑技能通过点击自己已经建好的建筑牌触发。
- 需要弃手牌的建筑技能会进入本地选择状态，在手牌右侧显示确认和放弃；只有确认时才向服务端发送牌 ID。
- 隐藏信息统一通过 `server/src/game/visibility.ts` 生成玩家视角状态，不能直接广播服务端完整 `GameRoom`。

### 结算界面

- 正式结算模块位于 `client/src/pages/result`。
- 获胜者会展示玩家头像和名称。
- 排名展示建筑分、奖励分和总分。
- 每位玩家的已建建筑会在结算中完整公开展示。
- 4 到 8 人排名区域采用纵向滚动，单个玩家建筑轨道可以横向滚动。

- 对局新增 `GameCardInspector.tsx` 与 `cardInspectorData.ts`，角色牌、自己的手牌、抽牌候选、自己已建建筑和其他玩家公开建筑统一支持鼠标悬停大卡预览。

## 6. 后续建议

下一次对话可以从以下事项继续：

1. 当前对局的防重叠、悬停检视器、右下操作台与 4 到 8 人位置布局已经完成阶段性验收；用户最新反馈是大致位置没有问题，但手牌区域、玩家标签等视觉比例仍难以匹配个人构想。后续不要整体推翻位置布局，优先按下方“UI 比例调音台”任务序列处理。
2. 后续正式卡牌美术生成或导入时，替换卡牌插图区，不改变当前卡牌容器比例与布局逻辑。
3. 后续如果引入新的角色或建筑牌，沿用当前显式目标选择与可见状态规则，不恢复默认目标。
4. 下一阶段可做正式版目录收口与无用测试页面清理，但必须先区分用户成果与真正废弃文件，不能批量删除未跟踪文件。

### 下一窗口任务序列：对局 UI 比例调音台

目标：保留已经通过验收的座位位置、安全区域和 4 到 8 人防重叠逻辑，把“组件放在哪里”与“组件看起来多大”分开，让用户可以直接调整手牌、玩家标签等视觉比例，再将满意参数固化为正式样式。

#### 第一阶段：先做可用的实时调参工具

1. 读取 `GameTableView.tsx`、`GameSelfArea.tsx`、`GamePlayerMiniStatus.tsx`、`GameOpponentSeat.tsx`、`game-table.css` 和 `game-opponents.css`，确认最终生效规则；不要先重写整个 CSS。
2. 在 `client/src/pages/game` 内新增开发专用的比例调节面板及配置模块，不把调参逻辑堆回 `GameTableView.tsx`。
3. 第一版只开放高影响参数：
   - 自己手牌的卡牌尺寸；
   - 手牌重叠量和手牌区域总宽度；
   - 玩家标签宽度、高度；
   - 玩家头像大小；
   - 对手身份牌、隐藏手牌和公开建筑牌大小；
   - 自己区域、建筑区、行动栏与手牌之间的垂直间距。
4. 参数通过对局根容器上的语义化 CSS 变量生效；卡牌优先使用单一宽度加 `aspect-ratio` 保持比例，避免分别调整宽高导致变形。
5. 面板仅在开发 / UI 验收模式显示，不能进入正式玩家界面；参数暂存到 `localStorage`，并提供恢复默认值与复制 / 导出当前配置的入口。
6. 增加“显示区域边界”开关，用半透明轮廓标出玩家名片、手牌、建筑区和行动区，便于用户判断实际占比。

#### 第二阶段：用户确认比例后再固化

1. 将用户确认的参数整理成对局专用视觉变量文件或清晰的对局样式变量区，避免同一尺寸继续散落在多个媒体查询中。
2. 响应式只保留标准桌面、中等桌面、低高度紧凑桌面三类必要修正；标准桌面以用户确认的视觉比例为基准，其余档位主要负责防重叠。
3. 清理被新变量替代的重复尺寸覆盖，但不要顺带改游戏规则、交互流程、可见状态或座位位置。
4. 如果新增文件或归类方式改变长期项目结构，同步更新 `docs/project-structure-rules.md`。

#### 验收顺序

1. 先在一个代表性视口（优先 1893×881、8 人）确认调节面板、实时变化、恢复默认和本地保存正常。
2. 用户确认视觉比例后，再运行现有轻量 UI 碰撞检查。
3. 固化样式后运行客户端构建，并复验 1365×668 低高度桌面及 4 到 8 人关键场景。
4. 第一阶段不要提前做完整 CSS 重构；目标是先让用户能快速表达并确定想要的比例。

## 7. 工作方式约定

- 默认由主 Agent 直接读取、修改和验证。
- 普通 UI / CSS 调整不使用 subagent。
- 只有真正独立、耗时较长或需要隐藏信息 / 安全审查的任务才启用 subagent。
- 单次最多 1 到 2 个 subagent，且不能同时写同一文件。
- UI 修改采用：读取相关模块 -> 主 Agent 修改 -> 单个代表性视口验收 -> 构建检查。
- 不重复扫描已确认范围，不为了形式分工增加等待时间和 token 消耗。
