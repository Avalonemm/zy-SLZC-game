阶段三单人测试方案
版本：v0.1
适用阶段：阶段 1、阶段 2、阶段 3
最后更新：2026-06-28

测试目标：
1. 确认项目可以启动和关闭。
2. 确认一个人可以用 4 个浏览器窗口模拟 4 名玩家。
3. 确认房间大厅流程可用：创建房间、加入房间、准备、开始。
4. 确认阶段三的服务端游戏数据结构可初始化。
5. 明确当前还不能测试的内容，避免误判。

当前阶段说明：
- 阶段 1 已完成：项目初始化。
- 阶段 2 已完成：房间大厅。
- 阶段 3 已完成：基础游戏数据结构。
- 阶段 4 未开始：还没有真正的角色选择界面、行动流程、拿金币、抽牌、建造、结算。

重要提醒：
- 阶段 3 的“游戏数据初始化”目前发生在服务端内部。
- 前端页面目前只展示到房间大厅和房间状态。
- 点击“开始游戏”后，房间状态会变成“已开始”，但还不会进入可操作的游戏桌面。
- 所以阶段 3 需要用自动测试确认初始化数据，而不是只靠前端页面观察。

准备工作：
1. 打开项目目录：
   I:\工作\ZY

2. 确认依赖已安装：
   如果已经能双击启动游戏.cmd，就说明依赖通常已经装好。
   如果不确定，可以在终端运行：
   npm install

测试 A：自动测试，推荐先跑
目的：
- 快速确认阶段 2 和阶段 3 的服务端核心逻辑没有坏。
- 这一步最能证明阶段 3 的数据结构初始化正确。

操作：
1. 在项目根目录打开终端。
2. 运行：
   npm run test --workspace server

期望结果：
- 看到 roomManager.test.ts 通过。
- 看到 gameSetup.test.ts 通过。
- 总测试数量应为 5 个通过。

这些测试覆盖了：
1. 创建房间后，房主自动进入房间。
2. 房间最多允许 4 名玩家。
3. 未满 4 人不能开始。
4. 有玩家未准备不能开始。
5. 4 人都准备后，房主可以开始。
6. 可以从 roles.json 读取 8 张角色牌。
7. 可以从 districts.json 读取建筑牌。
8. 可以初始化完整 GameRoom。
9. 每位玩家初始 2 金币。
10. 每位玩家初始 4 张手牌。
11. 每位玩家城市区为空。
12. 当前阶段进入 ROLE_SELECTION。

通过标准：
- 命令结束时没有红色失败信息。
- 显示 2 个测试文件通过，5 个测试通过。

失败时怎么处理：
- 如果提示找不到 npm：先安装 Node.js。
- 如果提示依赖缺失：运行 npm install。
- 如果某个测试失败：不要继续做阶段 4，先修复失败测试。

测试 B：完整构建测试
目的：
- 确认前端、后端、共享类型都能通过 TypeScript 和生产构建。

操作：
1. 在项目根目录打开终端。
2. 运行：
   npm run typecheck
3. 再运行：
   npm run build

期望结果：
- typecheck 通过。
- build 通过。
- client 会生成 dist。
- server 和 shared 的 TypeScript 检查通过。

通过标准：
- 两个命令都没有 error。

测试 C：一键启动和关闭测试
目的：
- 确认启动控件可以启动项目，并且关闭时会关闭服务器。

操作：
1. 双击项目根目录的：
   启动游戏.cmd

2. 等待浏览器自动打开：
   http://localhost:5173

3. 页面应显示：
   Socket 状态：已连接

4. 在启动窗口中按：
   Ctrl+C

5. 选择终止后，前端和后端服务应关闭。

通过标准：
- 页面能打开。
- Socket 状态显示“已连接”。
- 关闭启动窗口后，再刷新浏览器页面应无法继续连接服务。

测试 D：单人 4 窗口房间大厅测试
目的：
- 用一个人模拟 4 名玩家，验证阶段 2 的大厅流程。

操作：
1. 双击：
   启动游戏.cmd

2. 打开第 1 个浏览器窗口：
   http://localhost:5173

3. 第 1 个窗口输入昵称：
   Alice

4. 点击：
   创建房间

5. 记录页面显示的房间码，例如：
   ABCD12

6. 再打开第 2 个窗口，建议用无痕窗口或另一个浏览器标签页：
   http://localhost:5173

7. 第 2 个窗口输入昵称：
   Bob

8. 输入刚才的房间码，点击：
   加入房间

9. 第 3 个窗口重复加入，昵称：
   Cici

10. 第 4 个窗口重复加入，昵称：
   Dan

11. 回到 4 个窗口观察玩家列表。

期望结果：
- 4 个窗口都显示同一个房间码。
- 4 个窗口都能看到 Alice、Bob、Cici、Dan。
- Alice 显示为房主。
- Bob、Cici、Dan 显示为玩家。
- Alice 默认已准备。
- Bob、Cici、Dan 初始未准备。

测试 E：准备和开始测试
目的：
- 验证必须 4 人且全部准备后才能开始。

操作：
1. 沿用测试 D 的 4 个窗口。

2. 在 Bob 窗口点击：
   准备

3. 在 Cici 窗口点击：
   准备

4. 在 Dan 窗口点击：
   准备

5. 回到 Alice 窗口。

期望结果：
- Bob、Cici、Dan 状态变成“已准备”。
- Alice 窗口的“开始游戏”按钮可点击。

6. 在 Alice 窗口点击：
   开始游戏

期望结果：
- 房间状态变成“已开始”。
- 当前不会出现真正的游戏桌面，这是正常的。
- 因为阶段 4 还没开始制作。

通过标准：
- 未满 4 人时，房主不能开始。
- 4 人中有人未准备时，房主不能开始。
- 4 人都准备后，房主可以开始。
- 点击开始后，所有窗口同步看到状态变化。

测试 F：阶段三数据初始化确认
目的：
- 确认点击开始后，服务端确实初始化了 GameRoom。
- 这一步主要依靠自动测试，不依靠前端页面。

操作：
1. 在项目根目录打开终端。
2. 运行：
   npm run test --workspace server

重点查看：
- gameSetup.test.ts 是否通过。

它证明了：
1. 角色牌 JSON 能读取。
2. 建筑牌 JSON 能读取。
3. 服务端能初始化完整 GameRoom。
4. 玩家初始金币是 2。
5. 玩家初始手牌是 4 张。
6. 玩家城市区为空。
7. 游戏阶段进入 ROLE_SELECTION。
8. 建筑牌堆扣除了 4 名玩家共 16 张初始手牌。

当前不能通过前端确认的内容：
1. 不能在页面上看到自己的 4 张手牌。
2. 不能在页面上看到角色选择区。
3. 不能执行拿金币、抽牌、建造。
4. 不能结算分数。

这些会在阶段 4 和阶段 7 逐步加入。

测试 G：异常情况测试，可选
目的：
- 确认基础错误提示可用。

操作和期望：
1. 不输入昵称，创建房间按钮应不可用。
2. 输入不存在的房间码加入，应提示“房间不存在。”
3. 第 5 个玩家尝试加入同一房间，应提示“房间已满，最多 4 人。”
4. 非房主窗口不应该出现“开始游戏”按钮。
5. 房主窗口未满足条件时，“开始游戏”按钮不可用。

单人测试建议顺序：
1. 先跑测试 A：自动测试。
2. 再跑测试 B：完整构建。
3. 再跑测试 C：一键启动和关闭。
4. 再跑测试 D：4 窗口加入房间。
5. 再跑测试 E：准备和开始。
6. 最后看测试 F，确认阶段三的初始化能力。
