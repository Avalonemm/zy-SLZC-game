# 单人联机测试与重构检查清单

## 当前结论

当前项目已经可以继续做联机测试，不需要先做美术。

你已经在本机测试中成功完成过一整局，说明当前 MVP 的核心玩法链路已经跑通。后续重点不再是证明“能不能玩完”，而是把关键链路变成可重复检查，避免之后改规则、断线、UI 或技能时回归出问题。

但如果后续要加入倒计时、聊天框、动画、结算奖励、更多角色技能、2-8 人适配和 UI 资源替换，建议先整理几个高风险模块，避免所有功能继续塞进单个大文件。

## 模块分离现状

### 已经分离得比较好的部分

1. 共享类型
   - `shared/src/index.ts`
   - 前后端事件、房间、玩家、角色牌、建筑牌、游戏状态都有统一类型。

2. 建筑牌和角色牌数据
   - `server/src/data/roles.json`
   - `server/src/data/districts.json`
   - `server/src/game/cardData.ts`
   - 数据已经配置化，后续新增角色和建筑不需要全写死在前端。

3. 房间系统
   - `server/src/game/roomManager.ts`
   - 创建房间、加入房间、准备、开始、添加测试人机、断线重连基本独立。

4. Socket 入口
   - `server/src/socket/registerSocketHandlers.ts`
   - Socket 事件集中注册，能统一转发到服务端游戏逻辑。

5. 基础 UI 组件
   - `client/src/components/ui/*`
   - 按钮、卡牌、面板、输入框、玩家座位已经有基础组件。

6. 服务端规则辅助模块
   - `server/src/game/gameConfig.ts`
   - `server/src/game/gameEngineTypes.ts`
   - `server/src/game/gameEngineUtils.ts`
   - `server/src/game/visibility.ts`
   - `server/src/game/scoring.ts`
   - `server/src/game/roleSkills.ts`
   - 当前已先拆出配置常量、Result 类型、通用工具、隐藏信息过滤、结算逻辑和角色技能系统；`gameEngine.ts` 仍保留主要流程入口，避免一次性大重构。

7. 角色技能规则参考
   - `docs/角色技能规则设计表.md`
   - 用于后续设计进阶版技能、复杂目标选择、技能交互和可能的动画事件；当前只作为规则参考，不代表最终正式规则。

### 需要提前整理的部分

1. 游戏规则和状态机目前混在一个文件
   - 当前文件：`server/src/game/gameEngine.ts`
   - 当前进展：隐藏信息过滤已拆到 `visibility.ts`，计分结算已拆到 `scoring.ts`，通用工具已拆到 `gameEngineUtils.ts`，角色技能已拆到 `roleSkills.ts`。
   - 剩余风险：回合推进、基础行动和人机逻辑仍在同一个文件。
   - 后续建议继续拆分：
     - `turnFlow.ts`：阶段推进、回合推进、进入下一轮
     - `actions.ts`：拿金币、抽牌、建造、结束回合
     - `botPlayer.ts`：测试人机逻辑

2. 前端页面目前集中在一个大文件
   - 当前文件：`client/src/pages/ConnectionPage.tsx`
   - 风险：首页、房间页、游戏页、弹窗、头像、技能区、结算都在一起。
   - 建议拆分：
     - `HomePage.tsx`
     - `LobbyPage.tsx`
     - `GamePage.tsx`
     - `SkillPanel.tsx`
     - `PlayerStatusPanel.tsx`
     - `HandArea.tsx`
     - `CityArea.tsx`
     - `Scoreboard.tsx`

3. Socket 事件缺少服务层封装
   - 当前文件：`server/src/socket/registerSocketHandlers.ts`
   - 风险：倒计时、聊天、游戏提示、动画事件继续加入后会变长。
   - 建议拆分：
     - `roomSocketHandlers.ts`
     - `gameSocketHandlers.ts`
     - `chatSocketHandlers.ts`
     - `timerSocketHandlers.ts`
     - `broadcast.ts`

4. 人数规则目前仍有固定 4 人痕迹
   - 当前文件：`server/src/game/roomManager.ts`
   - 已有 `maxPlayers`，但 `MAX_PLAYERS = 4` 仍是固定常量。
   - 后续 2-8 人适配前，建议抽成 `GameConfig`：
     - `minPlayers`
     - `maxPlayers`
     - `endingCitySize`
     - `roleCount`
     - `initialGold`
     - `initialHandSize`

5. 动画和提示事件还没形成事件层
   - 当前主要靠 `game_state` 直接更新画面。
   - 后续动画建议增加服务端确认后的轻量事件：
     - `action_event`
     - `timer_event`
     - `chat_message`
     - `round_event`
     - `score_event`
   - 原则：动画只响应服务端结果，不决定游戏结果。

## 是否会造成大规模重构

如果现在继续直接往现有文件塞功能，会逐渐需要大规模重构。

如果先做上面的拆分，后续功能基本可以局部添加：

- 加倒计时：主要动 `timerSocketHandlers.ts`、共享类型、前端 `TimerBar`。
- 加聊天框：主要动 `chatSocketHandlers.ts`、共享类型、前端 `ChatPanel`。
- 加角色技能：主要动 `roleSkills.ts` 和 `roles.json`。
- 加结算奖励：主要动 `scoring.ts` 和前端 `Scoreboard`。
- 加 2-8 人：主要动 `GameConfig`、角色选择流程、座位布局。
- 换 UI 资源：主要动 CSS 和 UI 组件，不动规则。

## 单人怎么开始联机测试

### 方式 A：最快玩法测试，只用一个浏览器

适合测试完整一局，不需要真正模拟多个真人。

1. 双击 `启动游戏.cmd`。
2. 浏览器打开 `http://localhost:5173/`。
3. 输入昵称。
4. 点击“创建房间”。
5. 在房间页点击“添加测试人机”。
6. 想测 2 人局，就点 1 次。
7. 想测 3 人局，就点 2 次。
8. 想测 4 人局，就点 3 次。
9. 点击“开始游戏”。
10. 你自己选择角色。
11. 人机会自动选角和行动。
12. 你轮到行动时测试：
    - 使用技能
    - 拿金币
    - 抽牌
    - 建造
    - 结束回合
13. 打到自动结算。
14. 刷新页面，确认能恢复房间。

### 方式 B：单人模拟多个真人

适合测试真正的多客户端同步。

1. 双击 `启动游戏.cmd`。
2. 用 Chrome 打开 `http://localhost:5173/`。
3. 创建房间，记住房间码。
4. 再用 Edge 或 Chrome 无痕窗口打开同一个地址。
5. 输入另一个昵称，用房间码加入。
6. 如果要测 3-4 人，可以继续开无痕窗口，或者给房主添加测试人机。
7. 观察每个窗口的玩家列表、准备状态、房主转移、断线重连是否同步。

### 方式 C：只测 Socket 联机，不看 UI

适合快速验证服务端事件。

先双击 `启动游戏.cmd`，确认后端已经启动，然后运行：

运行：

```powershell
npm run smoke:reconnect
npm run smoke:local-flow
```

如果 `smoke:reconnect` 输出 `ok: true`，说明创建房间、断线、重连事件正常。

如果 `smoke:local-flow` 输出 `ok: true`，说明创建房间、添加测试人机、开始游戏和基础隐藏信息过滤正常。

## 当前最推荐测试顺序

1. 先用方式 A 打一局 2 人局。
2. 再用方式 A 打一局 4 人局。
3. 再用方式 B 开两个窗口，测试两个真人窗口同步。
4. 再用方式 B 测试隐藏信息：A 窗口不能看到 B 的手牌和未公开角色。
5. 刷新其中一个窗口，测试断线重连。
6. 当前阶段到这里即可验收；公网部署、租服务器、朋友远程加入都后置。

## 当前阶段验收结论

本机多身份测试也算联机测试，因为它仍然是真实的多个浏览器客户端通过 Socket.IO 连接同一个 Node.js 服务端，只是网络没有经过公网。

当前阶段不再要求 Render / Vercel / 云服务器部署。后续如果需要朋友远程测试，再优先考虑 Cloudflare Tunnel 临时公网方案；等游戏规则和 UI 都稳定后，再考虑正式云服务器。

当前已经完成一次本机完整对局，阶段 10A 的核心玩法验收可以视为通过。新增的两个 smoke 脚本用于后续每次修改后的快速回归检查，不能替代完整手动试玩，但可以提前发现连接、重连、开局和隐藏信息过滤这类基础问题。
